# CTF Write-up: Freebie

## Challenge Overview

**Category:** Web
**Objective:** Gain access to the classified documents on the `/flag` endpoint.

This challenge presents a secure system portal. Because the source code is not provided initially, the exploit requires a two-stage approach: first discovering a hidden debug parameter to leak the backend logic, and then forging a session cookie using an exposed secret key to bypass authentication.

---

## Step 1: Reconnaissance and Parameter Fuzzing

Initial inspection of the web application reveals standard login and registration endpoints, but no obvious vulnerabilities. When navigating a black-box web challenge, searching for hidden development or debug parameters is a standard initial vector.

Using a web fuzzer like `ffuf`, we can fuzz the URL parameters on the index endpoint to look for anomalous responses (such as different status codes or response lengths).

```bash
ffuf -w /path/to/wordlist/parameters.txt -u http://<target-ip>:5000/?FUZZ

```

The fuzzer reveals that supplying the `debug` parameter returns a significantly different response. Navigating to `http://<target-ip>:5000/?debug` in the browser triggers a backend hook that outputs the raw contents of `app.py`.

---

## Step 2: Source Code Analysis

With the `app.py` source code exposed, we can analyze the application's internal logic. Three critical details stand out:

1. **The Fuzzing Target Confirmed:** A `@app.before_request` hook explicitly checks if `"debug" in request.args`, and if so, opens and returns `__file__` (the current script).
2. **Admin Restrictions:** The goal is to hit the `/flag` endpoint, which requires the session user to be `admin`. However, both the `/register` and `/login` routes have hardcoded checks that explicitly prohibit creating or logging into the `admin` account via the web interface.
3. **The Vulnerability:** The application uses Flask's default client-side sessions. Critically, the developers left the secret key hardcoded in plain text at the top of the file:
`app.secret_key = "sup3r_s3cr3t_ctf_k3y_727"`

---

## Step 3: Session Forgery

Flask stores session data directly in the user's browser as a cookie. It relies on a cryptographic signature generated by the `secret_key` to ensure the data hasn't been tampered with. Because we now possess the secret key, we can forge our own valid session cookies.

We need to craft a session cookie claiming our username is `admin`. The tool `flask-unsign` handles this efficiently:

```bash
flask-unsign --sign --cookie "{'username': 'admin'}" --secret "sup3r_s3cr3t_ctf_k3y_727"

```

This command generates a signed session cookie string that the server will trust (e.g., `eyJ1c2VybmFtZSI6ImFkbWluIn0...`).

---

## Step 4: Retrieving the Flag

With the forged admin cookie ready, we can bypass the login mechanism entirely.

By injecting this cookie into our HTTP request headers, we can directly request the restricted endpoint:

```bash
curl -H "Cookie: session=<YOUR_FORGED_COOKIE_HERE>" http://<target-ip>:5000/flag

```

The server validates the cookie's signature using its secret key, decodes the payload, identifies the session as `admin`, and returns the HTML containing the flag.
